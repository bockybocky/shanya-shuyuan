# 山崖書院 — 設計歷程（2026-07-07，單日成形）

> 留這份檔的理由：沒有它，未來只看到「先考後讀＋雙欄鏡像」的結果，
> 看不到每個決定當時擋掉了什麼。

## 起源：兩個上游的合體

同一天遇到兩個東西：

1. **Garry Tan gbrain 的 `book-mirror`**——雙欄書鏡：左欄保書、右欄用讀者自己的個人資料庫寫人生對照；信任契約（書＝不可信輸入，逐章 agent 唯讀）；對讀者的事實查核（查無依據就刪）。
2. **Matt Pocock 的 teach skill**——教學法（ZPD／流暢度≠儲存強度／間隔與交錯）＋進度記憶（lessons/＋learning-records）。

使用者的合成指令：「有教學方法論＋會記錄教到哪＋按使用者資料產生個人化章節＋跟個人資料對照反思。**不做 app，就做 skill**——在 Claude Code / Codex 裡進行、產 HTML 互動閱讀。這比較像 agent 時代會出現的東西。」

關鍵前提：使用者的個人知識庫剛完成一輪深度自我盤點（行為模式、決策慣性的系統性挖掘）——右欄需要的個人脈絡密度正好現成。**沒有厚的個人脈絡，書鏡就只是普通摘要。**

## Grilling 五題與拍板（含一次反悔）

1. **互動形態**：先給三選項（課本式／對談式／混合）→ 使用者選混合 → **回上一題重問「還有其他可能性嗎」** → 加開四種（批註收割／先考後讀／行動驅動／聽讀）→ 拍板 **先考後讀＋行動驅動＋聽讀**。教訓：第一輪選項框太窄，學生要求擴散是好訊號，設計對談要留「回上一題」的路。
2. **章序**：教學重組（相關度×ZPD），棄「忠於原書」與「原書＋標記」——對「把書打到人身上」的目標，作者的鋪陳不如學生的相關度重要。
3. **個人料深度**：深層＋隱私閘——右欄可用最深的個人料，但音訊只含書內容、個人對照永留本機。
4. **試點書**：從使用者已有的自譯雙語書庫挑一本與其核心議題共振的。
5. **路徑**：先實跑不先建 skill → 試點跑到一半坑都踩完了，改令「完整規劃 skill」。

## 三個實戰事件如何改寫設計

### 事件一：「開卷 11 分鐘，人都走了」
試點時膠囊圖 agent 通讀全書花了約 11 分鐘，使用者一句「每一本書光打開第一頁就這麼久，人都走了」點出致命傷。
**解法（設計反轉）**：先考題考的是「既有直覺」，根本不用讀書——只需目錄＋導論。開卷 ≤2 分鐘變硬指標；備課在學生答題時背景平行。**學生的答題時間＝系統的備課時間**，死等變零。

### 事件二：「平時就先把書轉成 MD」→ 資料就緒原則
使用者追問「翻譯書的時候順便存 MD 是不是快多了」→ 產出三件事：
- `epub_to_md.py`（純標準庫秒級拆書，掛進書籍取得管線的固定一步）
- **膠囊拆兩層**：通用層（書摘/故事/金句，可預做永久有效）＋個人鉤子層（開課時對膠囊跑，便宜且隨人生變化重算）
- **原則升級**：「所有原料進系統必先轉成 agent 秒讀格式＋registry 登記＋定期巡檢」——書院的預處理需求升華成整個系統的資料紀律。

### 事件三：「怎麼不用 NotebookLM？」
使用者質疑 MD 轉換為何不用 NotebookLM → 釐清分層：**格式轉換**（機械、要忠實原文）NotebookLM 不適合（不回吐原文）；但**通用膠囊**正該走 NotebookLM RAG（引用式回答、帶頁碼、成本遠低於讓大模型通讀全書）。膠囊管線首選從「agent 讀全書」改為「NotebookLM 問答」。

## 被否決的方案（為什麼不是它們）

- **做 app**：明確否決——agent 時代的正解是 skill＋HTML，agent runtime 本身就是應用程式。
- **批註收割式**（HTML 表單匯出筆記）：沒被選為主軸，保留為未來可加的回流通道。
- **忠於原書章序**：見拍板 2。
- **膠囊全走大模型通讀**：成本量級差，NotebookLM RAG 勝出（fallback 保留）。

## 名字

**山崖書院**——《劍來》體系。同門還有問劍（驗訊號真偽）、論劍（驗預測記錄）：問劍驗因子、論劍驗人、**書院渡人**。
